How to Raise an IBM Support Ticket
Requesting Scheduled Weekly Restart — PA Cloud

Step-by-step guide to opening a Severity 4 service request asking IBM Cloud Ops to configure a recurring weekly TM1 instance restart.

Before You Start — What You Need
  • IBM ID registered at ibm.com/mysupport (same email linked to your PA Cloud contract)
  • IBM Customer Number (ICN) — from your purchase confirmation or your site administrator
  • Your PA Cloud database / instance name (e.g. Assurant_EPA_PRD)
  • Your PA Cloud tenant/environment URL (e.g. *.planning-analytics.saas.ibm.com)
  • Preferred restart time in UTC (e.g. Sunday 08:00 UTC = Sunday 04:00 ET)
  • Approval from your internal Change Advisory Board (CAB) / manager before submitting
1

Navigate to IBM Support Portal

URL → https://www.ibm.com/mysupport
Action
Go to mysupport.ibm.com and click "Open a case" (top navigation or dashboard tile). Sign in with your IBM ID. If first time, click "Register" and link your IBM Customer Number (ICN).
If you get "Access Denied", your IBM ID may not yet be linked to the PA Cloud contract. Ask your IBM site administrator to grant you access before raising a ticket.
2

Fill in the Case Header

Type of Support + Product Selection
Type of Support
Product Support
Product
IBM Planning Analytics (search for "Planning Analytics" and select)
Product Version
2.0.x / 2.1.x — select the version shown in your PAW Administration panel, or leave as "All Versions" if unsure
Operating System
Platform Independent (SaaS/Cloud)
3

Select the Correct Severity Level

IBM Enterprise Support — 4 severity levels
LevelIBM DefinitionUse When
SEV 1Critical — System Down. Production is completely unavailable. 24×7 engagement required.EPA instance unreachable for all users right now
SEV 2Significant. Severely restricted functionality or jeopardising a business deadline.Critical feature broken (e.g. all TI chores failing)
SEV 3Some impact. System usable; issue does not significantly affect operations.Non-critical bug or configuration question
SEV 4 ← USE THISMinimal — Inquiry / Non-technical request.Scheduling a recurring restart = enhancement/configuration request. System is currently running fine.
IBM explicitly says: Severity 4 = "An inquiry or non-technical request." A scheduled restart configuration is a proactive operational request — not an outage. Using SEV 1 or SEV 2 incorrectly can result in IBM downgrading your case and delaying response.
4

Write the Case Title

Keep it specific — IBM triages on title keywords
Recommended Title
[PA Cloud] Request: Configure Recurring Weekly Automatic TM1 Instance Restart for Memory Management — [Your Database Name]
Include your database name and "PA Cloud" in the title. IBM Cloud Ops teams are separate from on-prem support — this helps route directly to the right team.
5

Write the Case Description

Copy, fill in the bracketed fields, and paste into the IBM support form
CASE DESCRIPTION — COPY & PASTE
Subject: Request to Configure Recurring Weekly Automatic TM1 Instance Restart — IBM Planning Analytics Cloud Dear IBM Planning Analytics Cloud Support Team, We are writing to request that IBM Cloud Operations configure a recurring weekly automatic restart of our TM1 server instance on IBM Planning Analytics Cloud (SaaS). ───────────────────────────────────────── ENVIRONMENT DETAILS ───────────────────────────────────────── • Customer Name : Assurant Inc. • IBM Customer Number : [INSERT ICN] • PA Cloud Tenant URL : [INSERT e.g. https://xyz.planning-analytics.saas.ibm.com] • Database / Instance : [INSERT e.g. Assurant_EPA_PRD] • PA Version : [INSERT e.g. 2.0.9 / 2.1.x] • Environment Type : Production (IBM Planning Analytics Cloud — SaaS) ───────────────────────────────────────── BUSINESS JUSTIFICATION ───────────────────────────────────────── TM1's in-memory architecture does not release memory back to the operating system during runtime. Over a 7-day usage cycle, we observe progressive accumulation from: • Stargate / calculation view cache garbage • Abandoned sandbox accumulation • Zombie REST/PAX session threads • MUN lookup pre-computation bloat (PA 2.0.5+) • TurboIntegrator transient memory This results in degraded view open times, longer TI process runtimes, and occasional lock contention toward the end of each week. A weekly restart fully reclaims this memory and resets the platform to a stable baseline. We understand from the IBM Community (confirmed by IBM Offering Management — Stuart King) and customer precedent on TM1 Forum that IBM Cloud Operations can configure a Windows Scheduled Task on the backend VM to achieve this. Reference: https://community.ibm.com/community/user/businessanalytics/discussion/automated-restart-of-cloud-pa-server-instance ───────────────────────────────────────── REQUESTED CONFIGURATION ───────────────────────────────────────── • Restart Cadence : Weekly (every Sunday) • Preferred Time (UTC): 08:00 UTC (= 04:00 US Eastern / 03:00 US Central) • Action Sequence : Clean Stop → Wait for full shutdown → Start • Database Scope : Production instance only (details above) • Blackout Exception : Please skip on IBM's own 3rd-Saturday monthly maintenance week (to avoid double-restart) ───────────────────────────────────────── SPECIFIC QUESTIONS FOR IBM ───────────────────────────────────────── 1. Can IBM Cloud Operations configure this recurring restart as described? 2. What is the exact restart sequence IBM will use (SaveDataAll before stop?)? 3. Will we receive advance notification before each scheduled restart? 4. Is there a mechanism to temporarily suppress a specific week's restart (e.g., during month-end close) without raising a new ticket each time? 5. What monitoring / alerting does IBM apply to confirm successful restart and notify us if the restart fails or the instance does not come back up? ───────────────────────────────────────── NOTES ───────────────────────────────────────── • This is a proactive configuration request — the system is currently running normally. • We have confirmed via APAR PH19984 guidance that our tenant should be on PA 2.0.9+ before scheduling restarts (please confirm our current version). • We will subscribe to the PA Cloud status page for maintenance coordination: https://status.ai-apps-comms.ibm.com/planninganalytics We appreciate IBM's support in enabling this configuration. Regards, Niranjan Patra Senior TM1 / IBM Planning Analytics Consultant Tech Mahindra — Assurant EPA Engagement [Your contact email] | [Your phone]

Attach Supporting Documents

Optional but strongly recommended — speeds up IBM triage

🕐 What Happens After You Submit

📨
Immediate — IBM auto-acknowledgement email You receive a case number (e.g. TS012345678). Save this — use it in all follow-up communications.
👤
Within 2 business days — IBM Support Engineer assigned (SEV 4) They may ask clarifying questions about your environment. Respond promptly to avoid case going stale.
🔧
IBM routes to Cloud Ops team Support escalates to the IBM Cloud Operations team who manage the backend VMs. They review your requested cadence and time.
📅
IBM confirms schedule & tests first restart IBM will notify you of the first scheduled restart date and confirm the configuration. Verify your post-restart health probe fires correctly.
✅
Close the ticket + raise ServiceNow Standard Change Once IBM confirms the schedule is live, register it as a pre-approved Standard Change in Assurant's ServiceNow for SOX ITGC compliance.

⚠️ Common Mistakes to Avoid

Mistake Why It's a Problem Do This Instead
Raising as SEV 1 or SEV 2 IBM will downgrade immediately; delays assignment Always SEV 4 for proactive config requests
Vague title like "restart help" Routes to wrong team; slower triage Include "[PA Cloud]", database name, and "recurring restart" in the title
Not specifying time in UTC IBM Cloud Ops works in UTC; ET time causes confusion Always state UTC time. Add local time equivalent in brackets.
Forgetting blackout windows IBM may schedule restart during month-end close List all critical business blackout dates in the ticket
No internal approval before submitting SOX audit risk; change not in ServiceNow Get manager + CAB pre-approval; attach email to ticket
